# Skill 06 — 3D and CAD Asset Pipeline

## 1. Mission

**Separate mobile visualization assets from
authoritative engineering geometry** while
preserving semantic identity between them.

A GLB visual model is **not** equivalent to
production CAD. A `VISUAL_MESH` is a render-
time approximation; an `ENGINEERING_SOLID`
is the authoritative engineering geometry. A
pipeline that conflates the two is a contract
violation (per `.ai/AGENTS.md` section 5.1 +
`.ai/STANDARDS.md` section 2.1).

The pipeline is the bridge between the
user's 3D data (a STEP file from Onshape,
a glTF from Blender, a USD from Maya) and
the platform's typed world. The output is
a **canonical artifact** — a normalized,
content-addressed, signed, versioned
representation of the 3D data. The runtime
consumes the canonical artifact; it does
**not** consume the original CAD file.

## 2. Asset classes

The pipeline supports **11 asset classes**.
A class that is not in this list is a
contract violation; an ADR + a vote in
the AI council (skill 05) is required.

| Class | Purpose | Use |
|---|---|---|
| **`VISUAL_MESH`** | Render-time approximation | The mobile UX, the marketplace browse, the public 3D viewer |
| **`PARAMETRIC_MODEL`** | Parametric design | The forge UX, the designer's in-editor work |
| **`ENGINEERING_SOLID`** | Authoritative engineering geometry | The CAD server, the simulation, the homologation |
| **`COLLISION_MESH`** | Physics collision proxy | The field diagnostic flow, the simulation |
| **`PHYSICS_PROXY`** | Physics dynamics | The simulation, the digital twin (skill 07) |
| **`LOD`** | Level of detail | The mobile renderer (4 LODs per mesh) |
| **`TEXTURE`** | Image data | The renderer |
| **`MATERIAL`** | PBR material definition | The renderer |
| **`ANIMATION`** | Skeletal + blend shape | The renderer (animated components) |
| **`ASSEMBLY_MANIFEST`** | The semantic map | The renderer + the diagnostic + the simulation |
| **`ANNOTATION_LAYER`** | Text + measurement + highlight | The collaborative UX |

A `GLB visual model` is a `VISUAL_MESH`,
not a production asset. The classification
is enforced at the manifest level.

## 3. Mobile pipeline

The mobile pipeline is a **14-step pipeline**.
Every step is deterministic. Every step's
input is the previous step's output. A
step that fails halts the pipeline + emits
a typed `FoundryError`.

```
Source
  → virus and format scan
  → sandboxed parsing
  → unit normalization
  → coordinate normalization
  → semantic node mapping
  → mesh validation
  → polygon limits
  → texture limits
  → LOD generation
  → geometry optimization
  → KTX2 texture conversion
  → GLB packaging
  → manifest generation
  → hash
  → signature
  → publication
```

The per-step semantics:

1. **Source.** The original file
   (uploaded by the user, pulled
   from a supplier's catalog,
   generated by the AI council).
2. **Virus and format scan.** The
   file is scanned for known
   malware signatures + the
   format is detected from the
   magic bytes (per skill 03
   section 15 + the security
   skill 12's allowlist).
3. **Sandboxed parsing.** The
   file is parsed in a sandbox
   (no network, no file I/O
   outside the sandbox). A
   glTF with a script, a STEP
   with macros, a USD with a
   custom schema that runs
   code: rejected at this step
   (per `.ai/AGENTS.md` section
   5.5 + `.ai/STANDARDS.md`
   section 2.5).
4. **Unit normalization.** The
   asset's units are normalized
   to SI (per skill 03 section
   16). A `STEP` file in inches
   is rejected unless the
   importer can convert.
5. **Coordinate normalization.**
   The asset's coordinate
   system is normalized to
   right-handed Y-up (per
   `.ai/STANDARDS.md` section
   2.1). A left-handed Y-up or
   right-handed Z-up asset is
   transformed.
6. **Semantic node mapping.**
   The asset's nodes are mapped
   to `PartInstanceId`s +
   `AssemblyInstanceId`s (per
   section 4). A node without
   a semantic mapping is
   flagged.
7. **Mesh validation.** The
   meshes are validated
   (manifold, no T-junctions,
   no non-manifold edges).
8. **Polygon limits.** The
   polygon count is checked
   against the documented
   budget (default: 100k
   polygons per `VISUAL_MESH`).
9. **Texture limits.** The
   texture count + the texture
   size are checked against
   the documented budget
   (default: 4k × 4k, max 16
   textures per material).
10. **LOD generation.** Every
    mesh is decimated to 4 LODs
    (full / 75% / 50% / 25%).
    Each LOD is a separate
    artifact with its own hash.
11. **Geometry optimization.**
    Vertex welding + normal
    recomputation + tangent
    recomputation + Draco mesh
    compression.
12. **KTX2 texture conversion.**
    Every texture is converted
    to KTX2 with basis universal
    compression.
13. **GLB packaging.** The
    canonical scene is packaged
    as glTF 2.0 binary (`.glb`).
14. **Manifest generation.**
    The signed `SceneManifest`
    is generated (per
    `Spec.Artifact` from skill
    04).
15. **Hash.** The canonical
    artifact is content-
    addressed (SHA-256).
16. **Signature.** The
    canonical artifact is
    signed (Ed25519) by the
    producing agent.
17. **Publication.** The
    artifact lands in the
    content-addressed store +
    the metadata lands in the
    catalog (skill 09).

## 4. Semantic mapping

Every selectable object in the 3D scene
**MUST** reference a stable
`PartInstanceId` or `AssemblyInstanceId`
(per skill 03 section 13.4).

**Never** derive business identity from
mutable mesh-node names. A `mesh_104`
is a render-time handle; the business
identity is the `PartInstanceId`. A
`mesh_104` that is renamed in a future
revision of the GLB does **not** break
the business identity.

Example manifest:

```json
{
  "sceneId": "SCENE-POWERTRAIN-001",
  "vehicleRevisionId": "REV-42",
  "assets": [
    {
      "uri": "powertrain-lod1.glb",
      "sha256": "...",
      "sizeBytes": 18240000
    }
  ],
  "nodes": [
    {
      "renderNode": "mesh_104",
      "partInstanceId": "PARTINST-ENGINE-BLOCK-001"
    }
  ]
}
```

A node without a `partInstanceId` or an
`assemblyInstanceId` is a smell; the
node is decorative, not selectable.
A `renderNode` that has been renamed
must still resolve to the same
`partInstanceId` (per the manifest's
content hash).

A pipeline that emits a manifest
without semantic mapping is a contract
violation; the verifier (skill 14)
rejects the artifact.

## 5. Streaming (priority order)

The renderer streams assets in a
priority order. The order is:

1. **Vehicle shell first.** The
   outer body (the visual hull
   of the vehicle) loads first.
2. **Active subsystem second.**
   The subsystem the user is
   currently interacting with
   (the powertrain, the
   chassis) loads second.
3. **Detailed parts on
   selection.** A specific
   `PartInstance` loads when
   the user selects it.
4. **Internal fasteners only in
   repair mode.** The bolts,
   the washers, the nuts load
   only when the user enters
   the repair mode (per skill
   07).
5. **High-resolution textures
   only when needed.** A 4k
   texture loads when the user
   zooms in; a 1k texture is
   the default.
6. **Cancel obsolete loading
   requests when the user
   changes subsystem.** A
   request for the chassis
   details is cancelled when
   the user switches to the
   powertrain.

The streaming is **lifecycle-aware**:
a request is cancelled when the
lifecycle owner ends (per
`.ai/AGENTS.md` section 5.4).

## 6. Android constraints

The mobile renderer is a critical
surface. The following are
**forbidden**:

- **Decode large assets on the
  main thread.** Every
  decode is on `Dispatchers.IO`.
- **Retain duplicate GPU
  resources.** A texture is
  loaded once; subsequent
  requests share the same
  GPU resource.
- **Reload unchanged assets
  after recomposition.** A
  recomposition that does
  not change the asset does
  not re-decode the asset.
- **Create unbounded texture
  caches.** The cache size
  has a documented budget
  (default: 256 MB).
- **Keep obsolete scenes
  alive.** A scene that is
  no longer visible is
  disposed.
- **Render thousands of
  labels simultaneously.** A
  default zoom renders
  subsystem clusters; a
  selected item renders one
  primary label.

The following are **required**:

- **Lifecycle-aware renderer
  ownership.** The renderer
  is owned by the lifecycle
  owner (the `ViewModel`).
- **Explicit GPU disposal.**
  Every GPU resource is
  explicitly disposed; no
  reliance on the GC.
- **Memory-pressure
  handling.** A memory
  pressure event triggers
  a cache eviction.
- **Device capability
  profiles.** A low-end
  device gets the lowest
  LOD; a high-end device gets
  the highest LOD.
- **Dynamic LOD.** The LOD
  is adjusted per frame
  based on the camera
  distance + the frame time.
- **Frame-time
  instrumentation.** Every
  frame is instrumented (P50
  + P99 frame time + the
  GPU time + the draw
  call count).
- **Graceful degradation.**
  A device that cannot run
  the high-end renderer
  gets the low-end renderer
  + a "low graphics mode"
  warning.

A constraint violation is a CI
failure. The CI runs the
constraint test on every
release.

## 7. Marker and label policy

The 3D viewport renders markers +
labels. The policy:

- **At default zoom:** show
  subsystem clusters; hide
  individual labels; display
  one primary selected label.
- **Use occlusion-aware
  markers.** A marker that
  is occluded by a mesh is
  dimmed (not removed).
- **Avoid text overlap.** A
  label that overlaps another
  label is offset or
  suppressed.
- **Provide search-driven
  highlighting.** A user
  search ("battery pack")
  highlights the matching
  `PartInstance`s.
- **Support accessibility
  outside the 3D viewport.**
  A blind user uses the
  list view + the audio cues
  + the haptic feedback
  (per skill 11).

A label that violates the policy
is a contract violation; the
verifier (skill 14) rejects the
release.

## 8. CAD boundary

Engineering-solid operations
(`ENGINEERING_SOLID` + `PARAMETRIC_MODEL`)
**MUST** occur in a controlled
server-side compute environment
**unless** a supported native
engine exists (the engine is
allowlisted by skill 12).

Engineering exports require:

- **Valid closed solids.** A
  non-manifold solid is
  rejected.
- **Units.** The unit
  declaration (per skill 03
  section 16).
- **Materials.** The material
  declaration.
- **Tolerances.** The
  tolerance declaration
  (per `EngineeringFact<T>`).
- **Feature history or
  declared limitation.** A
  STEP file with a full
  feature history OR a
  declared limitation
  (e.g. "this STEP is
  tessellated; the feature
  history is lost").
- **Revision.** The
  `RevisionId` of the
  `PartDefinition` the
  geometry represents.
- **Provenance.** The
  `EngineeringFact<T>` with
  the full metadata (per
  `.ai/STANDARDS.md` section
  3).
- **Verification status.**
  The `VerificationStatus`
  of the geometry (per
  `.ai/STANDARDS.md` section
  3.2).

A engineering export without
these 8 fields is a contract
violation; the verifier rejects
the export.

## 9. Workflow

1. **Receive the upload.** The
   pipeline receives the
   original file + the
   `Spec.Artifact` from
   skill 04.
2. **Run the 14-step mobile
   pipeline** (per section 3).
3. **Generate the LODs +
   bounds + previews.**
4. **Sign + publish.** The
   artifacts are signed and
   stored in the content-
   addressed store.
5. **Report.** The
   `ProcessingReport` is
   emitted to the user.

## 10. Quality gates

- The original file is well-
  formed.
- The original file passes
  the validation suite
  (manifold + units +
  coordinate system + file
  size + no-embedded-scripts
  + provenance coverage).
- The canonical artifact is
  generated.
- The 4 LODs are generated.
- The bounds are computed.
- The previews are generated.
- The metadata is signed.
- The provenance is signed.
- The semantic mapping is
  complete (every selectable
  node has a
  `partInstanceId` /
  `assemblyInstanceId`).
- The mobile renderer
  satisfies the 6 Android
  constraints (per section
  6).
- The marker + label policy
  is satisfied (per section
  7).
- The engineering exports
  satisfy the CAD boundary
  (per section 8).

## 11. Definition of done

The pipeline is accepted only
when **every** proof below
passes.

- **Asset integrity before
  loading.** A test asserts
  the content hash + the
  signature + the provenance
  are verified BEFORE the
  asset is loaded into the
  GPU.
- **Stable semantic picking.**
  A test asserts a
  `partInstanceId` resolves
  to the same `partInstanceId`
  across a render
  recomposition.
- **No main-thread stalls
  above approved budget.** A
  test asserts every asset
  decode is on
  `Dispatchers.IO`; a
  main-thread decode is a
  CI failure.
- **No retained GPU resources
  after scene closure.** A
  test asserts every GPU
  resource is disposed on
  scene close.
- **LOD transitions are
  stable.** A test asserts
  the LOD transition is
  smooth (no popping + no
  flicker).
- **Malformed assets are
  rejected.** A test asserts
  every known malformed
  asset is rejected at the
  sandbox step.
- **Visual-only models are
  labeled accurately.** A
  test asserts every
  `VISUAL_MESH` is labeled
  as such in the manifest;
  a `VISUAL_MESH` that is
  labeled as
  `ENGINEERING_SOLID` is
  rejected.

## 12. Failure modes

- **The original file is
  corrupt.** The pipeline
  emits a `ProcessingReport`
  with the parse error.
  The user re-uploads.
- **The validation suite
  fails.** The pipeline
  emits a `ProcessingReport`
  with the violation + a
  fix suggestion.
- **The file is over the
  size budget.** The
  pipeline emits a
  `ProcessingReport` with
  the size + the budget.
  The user compresses or
  requests a budget
  exception (an ADR).
- **The original is in a
  non-supported format.**
  The pipeline rejects with
  a list of supported
  formats.
- **The pipeline itself
  fails** (e.g. Draco
  encoding crashes). The
  pipeline emits a
  `ProcessingReport` with
  the stack trace. The
  orchestrator escalates.
- **A `renderNode` is
  renamed** in a future
  GLB. The semantic map
  resolves the rename; the
  `partInstanceId` is stable.

## 13. Coordination contract

- **Input from**: skill 04
  (the spec), the user, the
  marketplace (skill 10).
- **Output to**: skill 07
  (digital twin), skill 09
  (catalog), skill 10
  (marketplace), skill 11
  (mobile renderer).
- **Triggered by**: every
  3D / CAD asset upload.
- **Frequency**: per asset,
  per revision.

## 14. Forbidden patterns

- **Storing the original +
  skipping the canonical.**
  The runtime consumes the
  canonical; the original is
  for the user's archive.
- **Unsigned artifacts.**
  Every artifact is signed.
- **Uncompressed meshes on
  mobile.** Every asset has
  at least 4 LODs.
- **Custom formats.** The
  canonical is glTF 2.0.
- **Missing provenance.**
  Every artifact has a
  provenance chain.
- **In-memory processing
  without streaming.** A
  100 MB asset is processed
  in a streaming pipeline.
- **Skipping the validation
  suite.** The suite is the
  safety net.
- **Trusting imported 3D
  assets.** An unvalidated
  asset is a contract
  violation.
- **Executing scripts
  embedded in uploaded
  assets.** A glTF with a
  script is rejected at the
  sandbox step.
- **Visual-mesh-is-not-
  mechanical-compatibility.**
  A mesh that visually
  matches is not
  mechanically compatible.
- **Promoting an AI-generated
  mesh to `OEM_EXACT` or
  `OEM_PARTIAL`.** A
  generated mesh is
  `PARAMETRIC_FUNCTIONAL`
  or `CONCEPTUAL`.
- **Deriving business
  identity from mutable
  mesh-node names.** The
  identity is the
  `PartInstanceId`.
- **Decode on the main
  thread.** Every decode
  is on `Dispatchers.IO`.
- **Retain duplicate GPU
  resources.** Every GPU
  resource is loaded once.
- **Render thousands of
  labels at default zoom.**
  A label policy violation.

## 15. Working with this skill

When invoked, this skill:

1. Receives the original file
   + the spec.
2. Runs the 14-step pipeline
   (per section 3).
3. Generates the LODs +
   bounds + previews.
4. Signs + publishes.
5. Reports to the
   orchestrator with the
   artifact IDs + a link to
   the user.

The skill does not render the
asset in 3D (skill 11). The
skill does not simulate it
(skill 07). The skill does not
list it (skill 10). The skill
produces the canonical
artifact; the next skills
consume it.
