# Skill 06 — 3D / CAD Asset Pipeline

## 1. Mission

Import, validate, normalize, content-address, sign,
and store every 3D + CAD asset the platform handles.
The pipeline is the bridge between the user's 3D
data (a STEP file from Onshape, a glTF from Blender,
a USD from Maya) and the runtime's typed spec.

The output of the pipeline is a **canonical
artifact** — a normalized, content-addressed,
signed, versioned representation of the 3D data.
The runtime consumes the canonical artifact; it
does not consume the original CAD file.

## 2. In-scope

- Importing from the supported formats: glTF 2.0,
  STEP (AP214 / AP242), USD, FBX (read-only).
- Validating: the asset is well-formed, the units
  are SI, the coordinate system is right-handed
  Y-up, the meshes are manifold, the materials
  are PBR.
- Normalizing: every asset is converted to the
  canonical format (glTF 2.0 + Draco mesh
  compression + KTX2 textures + basis universal).
- Content-addressing: the canonical artifact's ID
  is its SHA-256.
- Signing: every artifact is signed by the
  producing agent (the user's key for user-
  uploaded assets; the marketplace's key for
  marketplace assets).
- Storing: the artifact lands in the content-
  addressed store (S3-compatible). The metadata
  lands in the catalog (skill 09).
- LOD generation: every mesh is decimated to
  4 LODs (full / 75% / 50% / 25%). The LODs are
  content-addressed separately.
- Bounds computation: every mesh has a computed
  AABB + OBB + bounding sphere.
- Preview generation: every asset has a
  rendered preview (4 angles × 2 lighting
  setups) at 512x512.

## 3. Out-of-scope

- The rendering itself (skill 11 owns the
  mobile renderer; the web / desktop renderer
  is a follow-up).
- The simulation / digital twin (skill 07).
- The royalty calculation (skill 09).
- The marketplace (skill 10).

The pipeline produces a canonical artifact. The
marketplace lists it. The royalty engine
calculates on it. The digital twin simulates
it. The mobile renderer displays it. Each is
its own concern.

## 4. Inputs

- The original CAD file (uploaded by the user,
  pulled from a supplier's catalog, or
  generated by the AI council).
- The `Spec.Artifact` from skill 04 (the typed
  spec the asset is associated with).
- The ontology types (skill 03) — what type of
  asset is this? (A `BatteryPack` is rendered
  differently from a `Chassis`).
- The user's brand + project context.

## 5. Outputs

- A canonical artifact (glTF 2.0 + Draco + KTX2)
  in the content-addressed store.
- A signed metadata artifact (the manifest:
  hash, signature, units, bounds, LOD paths,
  material manifest, preview paths).
- A preview artifact (4 angles × 2 lighting
  setups at 512x512).
- A LOD manifest (4 LODs with their hashes).
- A processing report (errors, warnings, info
  notes).
- A provenance artifact (who produced the
  original, who imported it, when, with what
  toolchain).

The artifacts land in the content-addressed
store. The metadata + provenance land in the
catalog (skill 09).

## 6. Workflow

1. **Receive the upload.** The pipeline
   receives the original file + the
   `Spec.Artifact` reference.
2. **Identify the format.** Detect the format
   from the magic bytes + the file extension.
3. **Parse.** Parse the file into an internal
   scene graph (a typed `Scene` with `Mesh`,
   `Material`, `Texture`, `Light`, `Camera`
   nodes).
4. **Validate.** Run the validation suite:
   - Units are SI (reject files in inches /
     feet / custom units).
   - Coordinate system is right-handed Y-up.
   - Meshes are manifold (no T-junctions, no
     non-manifold edges).
   - Materials are PBR (no legacy Phong, no
     non-PBR workflows).
   - File size is within the documented budget
     (default: 100 MB per asset, configurable
     per project).
5. **Normalize.** Convert the scene graph to
   glTF 2.0. Apply Draco mesh compression. Apply
   KTX2 texture compression. Use basis
   universal for any non-PBR textures.
6. **Generate LODs.** Decimate each mesh to 4
   LODs. Each LOD is a separate artifact with
   its own hash.
7. **Compute bounds.** AABB, OBB, bounding
   sphere. The bounds are in the manifest.
8. **Generate previews.** Render 8 stills
   (4 angles × 2 lighting setups) at 512x512.
   Use the standard PBR + IBL setup (the same
   lighting the mobile renderer uses, so the
   preview is representative).
9. **Sign.** Sign the canonical artifact +
   the manifest + the provenance.
10. **Store.** Land the artifacts in the
    content-addressed store. Land the metadata
    + provenance in the catalog.
11. **Report.** Emit the processing report to
    the user.

## 7. Quality gates

- The original file is well-formed.
- The original file passes the validation
  suite.
- The canonical artifact is generated.
- The 4 LODs are generated.
- The bounds are computed.
- The 8 previews are generated.
- The metadata is signed.
- The provenance is signed.
- The artifacts are stored in the content-
  addressed store.
- The metadata is in the catalog.
- The processing report is in the user's
  inbox.

A failed gate is a blocker. The pipeline does
not produce a half-baked artifact.

## 8. Failure modes

- **The original file is corrupt.** The
  pipeline emits a processing report with the
  parse error. The user re-uploads.
- **The validation suite fails.** The
  pipeline emits a processing report with the
  violation + a fix suggestion. The user fixes
  the original + re-uploads.
- **The file is over the size budget.** The
  pipeline emits a processing report with the
  size + the budget. The user compresses the
  original + re-uploads, or requests a budget
  exception (an ADR).
- **The original is in a non-supported format.**
  The pipeline rejects the upload with a list
  of supported formats.
- **The processing pipeline itself fails**
  (e.g. Draco encoding crashes). The pipeline
  emits a processing report with the
  stacktrace. The orchestrator escalates.

## 9. Coordination contract

- **Input from**: skill 04 (the spec), the
  user, the marketplace (skill 10).
- **Output to**: skill 07 (digital twin), skill
  09 (catalog), skill 10 (marketplace), skill
  11 (mobile renderer).
- **Triggered by**: every 3D / CAD asset
  upload.
- **Frequency**: per asset, per revision.

## 10. Forbidden patterns

- **Storing the original + skipping the
  canonical.** The runtime consumes the
  canonical; the original is for the user's
  archive. Mixing the two is a contract
  violation.
- **Unsigned artifacts.** Every artifact is
  signed. An unsigned artifact is a rejection.
- **Uncompressed meshes on mobile.** Every
  asset has at least 4 LODs. The lowest LOD
  is the one shipped to mobile.
- **Custom formats.** "We use a binary
  format that nobody else can read" is a
  contract violation. The canonical is glTF 2.0.
- **Missing provenance.** Every artifact has a
  provenance chain. An asset without provenance
  is rejected.
- **In-memory processing without streaming.**
  A 100 MB asset is processed in a streaming
  pipeline, not loaded into memory all at once.
  Out-of-memory is a quality-gate failure.
- **Skipping the validation suite.** The
  validation suite is the safety net. A
  pipeline that skips the suite is a violation.

## 11. The pipeline in the Elysium Automotive
Foundry

The pipeline is the bridge between the user's
3D world and the platform's typed world. A
supplier uploads a STEP file of a battery pack;
the pipeline converts it to glTF 2.0, generates
4 LODs, computes the bounds, renders 8 previews,
signs the manifest, and stores the result. The
marketplace lists the asset. The digital twin
simulates the part. The mobile renderer displays
it. The royalty engine calculates on it.

The pipeline is the platform's "what does this
3D data look like, in our world?" answer.

## 12. Working with this skill

When invoked, this skill:

1. Receives the original file + the spec.
2. Parses + validates + normalizes.
3. Generates the LODs + bounds + previews.
4. Signs + stores.
5. Reports back to the orchestrator with the
   artifact IDs + a link to the user.

The skill does not render the asset in 3D
(skill 11). The skill does not simulate it
(skill 07). The skill does not list it
(skill 10). The skill produces the canonical
artifact; the next skills consume it.
