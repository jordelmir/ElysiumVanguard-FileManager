# ADR-010: Application capsule format and registry

- Status: Accepted
- Date: 2026-07-13
- Owners: Elysium Vanguard runtime
- Depends on: ADR-001

## Context

Users need a way to install, organize and launch different types of
applications — Linux native, PRoot-managed, Windows (via Wine), remote services
and virtual machines — without manually configuring the runtime backend each
time. The app needs a uniform packaging format that describes what an
application is, what it needs and how to run it.

## Decision

Define an ApplicationCapsule as the universal packaging format:

### Capsule format (JSON)

```json
{
  "capsuleVersion": 1,
  "id": "com.example.myapp",
  "name": "My Application",
  "version": "1.0.0",
  "type": "linux_proot",
  "icon": "base64-encoded PNG",
  "executable": "/opt/myapp/bin/start.sh",
  "workingDirectory": "/opt/myapp",
  "environment": {
    "DISPLAY": ":0",
    "MYAPP_CONFIG": "/host/config/myapp.conf"
  },
  "capabilitiesRequired": ["DISPLAY", "NETWORK"],
  "permissions": [
    {"resource": "STORAGE", "access": "READ_WRITE", "path": "/host/sdcard"},
    {"resource": "NETWORK", "access": "OUTBOUND"}
  ],
  "resources": {
    "minMemoryMb": 256,
    "recommendedMemoryMb": 512,
    "minDiskMb": 100
  },
  "integrity": {
    "algorithm": "SHA-256",
    "hash": "abc123..."
  },
  "metadata": {
    "author": "Developer Name",
    "homepage": "https://example.com",
    "categories": ["Development", "Editor"],
    "description": "A description of the application"
  }
}
```

### Capsule types

| Type | Description | Backend |
|---|---|---|
| `linux_native` | Native ARM64 Linux binary | PRoot or direct |
| `linux_proot` | Linux binary with PRoot filesystem | PRoot |
| `windows_wine` | Windows PE binary via Wine+Box64 | WinLayer |
| `remote` | Service accessed over network | RemoteBackend |
| `vm` | Full virtual machine | AVF/QEMU |

### Registry

An in-memory CapsuleRegistry provides:
- CRUD operations for installed capsules.
- Filtering by type, capability and category.
- Search by name and description.
- Integrity verification against stored hashes.

### Storage

Capsules are stored as JSON files in `{appData}/capsules/{capsuleId}/capsule.json`.
The registry is rebuilt on app start by scanning the capsules directory.

### Integrity

Every capsule has an integrity hash covering the executable, support files and
manifest. Verification is performed before launch. A mismatch produces
`LaunchError.IntegrityFailure`.

## Invariants

1. Every capsule has exactly one manifest file.
2. Capsule IDs are reverse-domain-name unique.
3. Integrity verification runs before every launch.
4. The registry does not cache capsule files; it reads from disk on every
   query.
5. Capsules with unknown or future `capsuleVersion` are rejected.

## Alternatives considered

### Android-style APK packaging

Rejected. APK is specific to Android apps and requires Android tooling to
build. Capsules use a simple JSON format that can be generated by any CI
pipeline or manually by users.

### Flatpak/AppImage format passthrough

Rejected. Flatpak and AppImage are desktop Linux formats with no Android
adaptation. Capsules wrap these formats with runtime configuration needed for
Android.

## Consequences

- Users can distribute apps as capsule JSON + executable tarballs.
- The registry enables app-grid and launcher UIs without backend-specific
  branching.
- Integrity verification adds launch latency.
- Capsule format v1 is immutable; future versions will be additive only.
- A capsule SDK may be published to help developers create capsules.

## Revisit triggers

- Capsule format limits expressiveness for a new backend type.
- JSON parsing or file I/O becomes a performance bottleneck at >100 capsules.
- A community standard emerges for Linux mobile application packaging.
