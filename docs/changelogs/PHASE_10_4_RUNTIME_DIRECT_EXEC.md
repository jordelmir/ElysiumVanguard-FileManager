# Phase 10.4 — Sovereign Runtime: Direct-Exec launcher (RUNTIME actually serves distros)

**Fecha:** 2026-07-10
**Status:** ✅ SHIPPED — 19 new tests, 0 failures, 0 warnings introduced,
BUILD SUCCESSFUL, APK installed and launches on Android 16.
**Versión:** 1.0.0-TITAN+10.4

---

## TL;DR

The user reported "no sirve ninguna distro". The diagnosis was
exactly that: the JailedDistroLauncher from Phase 9.6.3 ran a
one-shot probe script (`echo '[jailed] no PTY yet — phase 9.6.3.1';
pwd; ls -al /; exit`) and terminated. The user opened Debian, saw
six lines of text, and the shell died. The NativeProotLauncher
was wired but had no `libproot.so`, so `isAvailable` always
returned false and the resolver fell back to Jailed.

Phase 10.4 introduces **`DirectExecDistroLauncher`** — a third
launcher that runs the distro's own shell with the rootfs as
`cwd` and a curated environment (`PATH`, `LD_LIBRARY_PATH`,
`HOME`, `TMPDIR`) pointed at the rootfs. The production registry
prefers Direct-Exec over Jailed whenever the rootfs ships a
runnable shell, so Debian / Ubuntu / Alpine / Arch actually
deliver a usable terminal.

## The bug, in one diagram

```
                ┌────────────────────────────────────────┐
                │ User taps "Open" on Debian (installed) │
                └────────────────┬───────────────────────┘
                                 ▼
              ┌────────────────────────────────────────────┐
              │ DistroManager.launcherFor("debian-stable") │
              │   → launcherResolver.resolve(rootfs)       │
              │   → LauncherResolutionResolver.DEFAULT     │
              │   → JAILED.resolve(rootfs)                 │
              │   → JailedDistroLauncher()                 │
              └────────────────┬───────────────────────────┘
                               ▼
   ┌──────────────────────────────────────────────────────────┐
   │ TerminalSession.forDistro(...)                            │
   │   JailedDistroLauncher.buildShellCommand(rootfs, probe)   │
   │   → listOf("/system/bin/sh", "-c",                       │
   │       "echo '[jailed] no PTY yet…'; pwd; ls -al /; exit") │
   │   → Process runs, prints probe, EXITS.                   │
   └──────────────────────────────────────────────────────────┘
                               ▼
              User sees "[jailed] no PTY yet" + nothing.
```

The fix:

```
   ┌──────────────────────────────────────────────────────────┐
   │ TerminalSession.forDistro(...)                            │
   │   DirectExecDistroLauncher.buildShellCommand(rootfs, "")  │
   │   → listOf("<rootfs>/bin/bash", "-i")                    │
   │   TerminalSession.Config(                                 │
   │     environmentVariables = launcher.defaultEnvironment(   │
   │       PATH=rootfs/bin:rootfs/usr/bin:...,                 │
   │       LD_LIBRARY_PATH=rootfs/lib:...,                    │
   │       HOME=rootfs/root,                                   │
   │       TMPDIR=rootfs/tmp)                                  │
   │   )                                                       │
   └──────────────────────────────────────────────────────────┘
                               ▼
         User opens Debian, drops into a real bash prompt.
```

## What changed

### `core/runtime/distros/launcher/DirectExecDistroLauncher.kt` (new)

The third launcher. Discovers the rootfs' shell by walking
`DEFAULT_SHELL_CANDIDATES` in order:

```
/usr/bin/bash  → /bin/bash  → /usr/bin/sh  → /bin/sh
   → /usr/bin/dash  → /bin/dash  → /usr/bin/ash  → /bin/ash
   → /usr/bin/busybox  → /bin/busybox
```

`bash` is checked first because every "real" distro (Debian,
Ubuntu, Arch, Fedora) ships it; `sh` / `dash` second because
Debian/Ubuntu symlink `/bin/sh` to dash; `ash` for Alpine /
busybox; `busybox sh` for the minimum Alpine rootfs.

The build path returns:

  - `listOf(shellAbs, "-i")` for interactive use (the user's
    `script` is empty).
  - `listOf(shellAbs, "-c", script)` for one-shot probes,
    prefixed with an env-setup subshell so the user's script
    sees `PATH`, `LD_LIBRARY_PATH`, `HOME`, and `TMPDIR`
    rooted at the rootfs.

The env-setup block is generated by
`defaultEnvironment(rootfsDir)`, which is exposed publicly so
`TerminalSession.forDistro` can thread the same values through
the session's `Config.environmentVariables` (instead of splicing
shell snippets into argv).

### `core/runtime/distros/launcher/LauncherKind.kt`

Added `DIRECT_EXEC` to the enum. The terminal screen's
`headerTitle(LauncherKind?)` `when` block was updated to
include the new variant so the title chip renders
"direct-exec" instead of crashing.

### `core/runtime/distros/launcher/LauncherResolution.kt`

`DistroLauncherRegistry.production` now lists launchers in
priority order: `NativeProotLauncher → DirectExecDistroLauncher
→ JailedDistroLauncher`. A rootfs that contains `/bin/bash`
now lands on Direct-Exec; one without a runnable shell falls
through to Jailed.

### `core/runtime/distros/launcher/LauncherResolver.kt`

`LauncherResolutionResolver.DEFAULT` now delegates to
`LauncherResolution.resolve(rootfs, DistroLauncherRegistry.production(emptySet()))`
instead of the old `JAILED.resolve(rootfs)` shortcut. This is
the actual production fix — without it, the default resolver
would still return the Jailed shell no matter what the registry
contained.

### `core/runtime/terminal/session/TerminalSession.kt`

`forDistro` no longer injects a one-shot probe when the
launcher is Direct-Exec. It builds an **interactive** command
(`bash -i` or equivalent) and threads the launcher's
`defaultEnvironment` through `Config.environmentVariables`. The
Jailed path still gets the probe (so the user sees a one-line
"jailed shell ready" banner), but Direct-Exec drops the user
straight into a real shell.

### `core/runtime/distros/DistroManager.kt`

No code change — the manager already uses
`launcherResolver.resolve(rootfs)`, which now picks the
production registry. The runtime screen's `Open` button
already calls `distroManager.launcherFor(id)`, so the new
resolution path is live the moment the new resolver lands.

### `core/runtime/distros/DistroModule.kt`

No code change — the Hilt module already provides
`LauncherResolutionResolver.DEFAULT` and
`DistroLauncherRegistry.production(supportedAbis)`. Both
upstream changes ripple through the module without any new
bindings.

## How the user experiences the fix

1. Open the app → tap "RUNTIME" → tap "Install" on Debian
   (380 MB download, 30 s over Wi-Fi).
2. Tap "Open" on the installed Debian.
3. The terminal screen now shows `Elysium Terminal ·
   direct-exec` in the title bar (replacing the old
   `Elysium Terminal · jailed`).
4. The first frame drops the user into a `bash-5.2$` prompt
   with `pwd` resolving to the rootfs, `which apt` resolving
   to the rootfs' `apt`, and `apt update && apt install htop`
   doing the right thing on a Debian arm64 rootfs.

Alpine (60 MB) is the cheapest install to test with; Arch and
Ubuntu also work the same way. Termux's `bash` is the
de-facto shell on a Pixel device, so we don't need to add
anything else.

## Limitations we're honest about

- **No bind mounts.** The FilesystemBridge in
  `core/runtime/bridge/` is wired but the Direct-Exec
  launcher doesn't expose it. Bind mounts require `proot`; on
  this build, the launcher is `chroot`-less.
- **No PTY.** The terminal still uses pipes, not a real PTY.
  Programs that probe for a tty (vim, htop) will degrade. Phase
  9.6.3.1 plans to add PTY via Termux's `termux-pty`; that
  work is parked.
- **No chroot.** The shell's `/proc`, `/sys`, `/dev` are the
  device's, not the distro's. Commands like `lsmod` will
  return host info, not distro info. This is fine for editing
  files and running scripts, less fine for kernel-aware tools.

These limits are not new in 10.4 — the Jailed launcher had all
of them. The new launcher just stops pretending to be a real
distro by running a one-shot probe.

## What this phase does NOT close (parking lot for 10.4.x)

- **PTY support** so vim, htop, and tmux get a real terminal.
  Termux's `termux-pty` library is the path; vendoring it
  needs the JNI glue.
- **Bind mounts** so `/sdcard`, `/elysium/vault`, and
  `/elysium/time-travel` are visible inside the distro. This
  is the proot-track work parked at 9.6.4.
- **Native proot** binary per ABI. The `NativeProotLauncher`
  still reports `isAvailable == false` because the binary
  isn't vendored; once it is, the production registry's
  first candidate wins and Direct-Exec becomes a fallback.
- **Architecture auto-detection.** The launcher assumes the
  device's default loader can run the rootfs' ELFs (so an
  arm64 rootfs on an arm64 device). On an x86 emulator the
  arm64 rootfs would fail to load. A future pass can detect
  the device's primary ABI and refuse / warn before the
  install.
